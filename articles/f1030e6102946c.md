## はじめに
自分が参画している現場で、GitHub Copilotを用いて、ローカルでAIレビューする仕組みを作ってみたので、どのようなものを作ったのかを紹介していきます。
GitHub Copilotも知らないうちに結構進化していて、VScodeで使う分には使いやすいと感じたので、「こういう部分が便利！」みたいな情報も合わせて伝えられればと思います。
Claude Codeなどと比べて、GitHub Copilotの機能などを紹介している記事は少ないように思うので、GitHub Copilotしか使えない環境の人にとって参考になる記事になれば幸いです。

:::message
仕事で作ったものと同じような仕組みをイメージして、プライベートPCで作っていますが、レビュー観点などは実際のものと異なり、AIに書かせたりしているので、かなり適当です。
また、GitHub Copilotも進化が早く、VScodeや拡張機能のバージョンが違った場合に、この記事で紹介したことと実態が異なる可能性がある点にはご注意ください。
:::

## /select_reviewコマンドについて
まず、選択範囲をレビューするスラッシュコマンド（`/select_review`）を作成していきます。これは少ない範囲でレビュー観点に合った実装ができているかを気軽にチェックするためのコマンドとして作成し、後述する`/review_before_pr`コマンドよりかシンプルな作りのため、先に解説します。
### prompt filesを使ってスラッシュコマンド作成
まず、GitHub Copilotでもagentやskillsなどいろんなものがあって、訳がわからなくなることがあると思います。今回全てを紹介しませんが、それぞれの違いについて知りたい人は、GitHub公式だと以下ドキュメントが参考になると思います。
https://docs.github.com/en/copilot/reference/customization-cheat-sheet
![](/images/f1030e6102946c/2.png =500x)
*参照リンクからの抜粋画像*

その中で、`.github/prompts`ディレクトリの中に`*.prompt.md`というファイルを作成することで、スラッシュコマンドを作成できます。
今回は、`select_review.prompt.md`を作成することで、ファイル名の前半部分と同じ`/select_review`というコマンドが実行できるようになります。
![](/images/f1030e6102946c/3.png =500x)
スラッシュコマンドは画像のようにcopilotの質問入力欄に`/`および文字を入力すると、候補が表示されて、簡単に実行することができます。
### select_review.prompt.mdの中身
実際の`select_review.prompt.md`は以下のようになっています。
```md:select_review.prompt.md
---
agent: agent
description: 現在選択しているコード範囲をレビューするよ
argument-hint: コードを選択してからこのプロンプトを実行してください
tools: [read, search, todo]
model: Claude Sonnet 4.6 (copilot)
---

# select_review
現在エディタで選択されているコード範囲について、このプロジェクトの規約・品質基準に沿ったコードレビューを行います。
```

プルリクエスト（PR）提出前に実行し、現在いるブランチからマージ先のブランチ(main)との差分を取得&AIレビューを実施することができるコマンドです。
### review_before_pr.prompt.mdの作成
まずは`/select_review`同様、.github/promptsディレクトリに以下ファイルを作成しています。
先ほどと異なる部分に着目して、説明していきます。

```md: review_before_pr.prompt.md
---
agent: agent
description: mainブランチとの差分を取得してコードレビューを行う
argument-hint: 変更内容の説明（目的・範囲など）を入力してください
tools: [execute, read, agent, search, todo]
model: Claude Opus 4.6 (copilot)

# コードレビュースキル

以下2点を満たすコードレビューを行います。

- linterの実行結果から修正すべき内容をまとめる
- `main` ブランチとの差分を確認し、このプロジェクトの規約・品質基準に沿ったコードレビューを行う

## ワークフロー

### Step 1: linterの実行とレビュー

`#tool:agent/runSubagent` を使用して、linter_review サブエージェントを呼び出す。

- 対象エージェント: linter_review.agent.md

### Step 2: AIレビュー

`#tool:agent/runSubagent` を使用して、ai_review サブエージェントを呼び出す。

- 入力: ユーザーからの変更内容の説明（目的・範囲など）が存在すれば、それを入力として使用。
- 対象エージェント: ai_review.agent.md

## Step 3: レビュー結果のまとめ

各エージェントからの出力を受け取り、レビュー結果の要約とファイル名をユーザーに報告してください。

```
### 処理フローの違い
まず、プロジェクトの都合により、Linterの実行結果のレビューとAIレビューの結果をそれぞれ1つのコマンド実行で確認できるようにしたかったという背景があります。AIレビューの結果もgit diffでコード差分を取得してくるためコンテキスト量が多くなることが想定され、さらにLinter結果をまとめるのも1つのセッションで実行するとコンテキストを圧迫し、精度が下がる可能性があると思いました。そこで、メインエージェント（`review_before_pr`）から、Linter実行結果をまとめるサブエージェントとコード差分からAIレビューを実行するサブエージェントにタスクを委譲し、最終結果の要約のみをまとめて、copilotチャット画面に表示するようにしました。
要約のみをチャット画面には表示しますが、LinterおよびAIレビューの結果はそれぞれファイル出力させることで、詳細を確認したり、必要な部分をcopilotに読ませて修正することが可能にしています。

## サブエージェントの作成
以下のドキュメントはカスタムエージェントについてですが、サブエージェントの場合もカスタムエージェントと同様の作り方となります。`.github/agents`ディレクトリに`*.agent.md`ファイルを作成する感じですね。
https://docs.github.com/ja/copilot/how-tos/use-copilot-agents/coding-agent/create-custom-agents
自分が過去に書いた以下の記事でもカスタムエージェントについては説明しています。
https://zenn.dev/peishim/articles/d6e44076d3fa0b

今回作ったAgentについて、詳しくはこの記事の最初に貼ったGitHubのページからagentsディレクトリの中のファイルを確認してもらえればと思いますが、「mainブランチとの差分を取得し、レビューするエージェント」の方に絞り、要点だけ説明します。

### frontmatter部分
以下がfrontmatterの設定となっています。
```md
description: mainブランチとの差分を取得してコードレビューを行う
tools: [execute, read, agent, edit/createFile, edit/editFiles, edit/rename, search, todo]
model: Claude Sonnet 4.6 (copilot)
user-invocable: false
```
prompt filesの時と同じような設定ですが、`user-invocable: false`というものを追加しています。エージェントファイルを作成すると以下のようにカスタムエージェントとして質問入力欄から選択できるようになります。自分の画面でも余計なエージェントが色々登録されてしまっていますが、サブエージェントとしてのみ動かしたい場合は、ここに表示させる必要がありません。`user-invocable: false`を設定することで、質問入力部分でカスタムエージェントとして表示されず、サブエージェントのみで扱えるようになります。
![](/images/f1030e6102946c/10.png =500x)

### コマンド実行
```bash
git diff origin/main...HEAD --name-only -- '*.py' '*.ts' '*.tsx' '*.css' '*.html'
```
prompt filesでも実行できますが、このようにコマンドを実行してほしいように指示を書けば、AIがコマンドを実行してくれます。上記は決まったコマンドですが、AIが柔軟にコマンドを修正して実行することもできるので、ファイルごとにgit diffするみたいなことも指示に書いてたりはします。ただ、コマンド実行のためにtoolsの`execute`は必要です。

## まとめ
今回紹介した内容をまとめると以下の通りです。

- `prompt files`（`.github/prompts/*.prompt.md`）を使えば、特定のユースケース向けのスラッシュコマンドを手軽に作成できる
- サブエージェントにタスクを委譲することで、コンテキストを分散させながら複数の処理を1コマンドで実行できる
- ツール設定やファイル参照などをGUIでサポートしてくれる仕組みがあり、設定ミスを防ぎやすい

読んでいただきありがとうございます。GitHub Copilotでできることもかなり増えているので、Copilotしか使えない環境でも、うまく活用すればいろんなことを自動化できそうと思っています。特にVS Codeと連携してエージェント等の定義や動作デバッグも便利にできそうなので、Claude CodeやCursorなどが強い中でもGitHub Copilotを使う価値もありそうだなと自分は感じています。
:::message
以下は本編に書くと少し冗長になると思ったおまけですので、良かったら読んでください。

## おまけ
### skillsでもスラッシュコマンドが作れる話
今回はprompt filesという機能を使ってスラッシュコマンドを作りましたが、skillsを作成してもスラッシュコマンドとして使用できます。なのでskillsを用いる選択肢もありました。skillsは毎質問でそのskillを使用するかどうかの判断をAIにさせることになりますが、レビュー機能は意図的に呼び出したい場合のみに動作させたかったため、prompt filesを活用しました。
後で調べたところ、skillsも特定のファイルをスラッシュコマンド以外からは読み込まれないようにするオプションを設定できるようだったのですが、同じことを実現するなら、skillsディレクトリに入っているファイルは自動で読み込まれる、promptsディレクトリはスラッシュコマンドのみという分け方で保存されている方がわかりやすいかと思い、今の仕組みにしています。
skillsで設定できるオプションは以下に記載されており、`disable-model-invocation`を`true`にすると自動でskillが読み込まれなくなります。
https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference#skills-reference
以下記事のようにClaude Codeではカスタムスラッシュコマンドがskillsに統合されたので、GitHub Copilotでも同じような道を辿る可能性はありますね。
https://zenn.dev/tmasuyama1114/articles/cc_commands_merged_into_skills

### コマンドの承認が頻繁に求められる場合
毎回承認を求められる場合は、まずAllowの右の三角からEnable Auto Approveをクリックします。
![](/images/f1030e6102946c/11.png =500x)
すると下画像のように通知モーダルが表示されます。
![](/images/f1030e6102946c/12.png =500x)
これはおそらく、ユーザー設定やプロジェクト設定で許可したコマンドのみを自動承認するためのEnableであり、これを設定した上で、以下のような特定のコマンドを許可する設定をワークスペースやユーザー設定に入れれば、承認なしで自動実行してくれると思います。
```json: .vscode/settings.json
  "chat.tools.terminal.autoApprove": {
    "git diff" : true,
    "ruff": true,
    "npx biome": true,
  }
```
ただ、エージェントがよしなにコマンドを作成＆実行したがるので、たまに許可を求められることもあります。その場合は、許可するコマンドの種類を増やすか、プロンプトで調整するしかないのかなと考えています。

:::message
確かな情報が確認できなかったこともあり、推測している部分もあります。ご自身の責任で設定するようにお願いします。自動承認は100%守られることを保証しておらず、そのために通知モーダルでbest effort protectionと書かれていると思います。
ツールの承認に関しては以下のようなドキュメントもあります。
https://code.visualstudio.com/docs/copilot/agents/agent-tools
:::

### スラッシュコマンド実行するとモデルやtoolsは設定した値に変わる
当たり前のことではあるんですが、スラッシュコマンドを実行するとスラッシュコマンドに設定しているモデルやツールに設定が変更されます。なので、モデルやツールがいつの間にか切り替わっていることに気づかない人もいるかもしれないという点には注意です。

### .github、.agents、.claudeなどの管理が大変問題
例えばGitHub Copilotにおいて、skillsは.githubディレクトリだけでなく、.agentsや.claudeディレクトリでも動作するようです。自分もclaude codeも使ったりするように、チームでもメンバーによって使うコーディングエージェントが異なる場合はこれらのファイルの管理は非常に面倒だなと感じています。正直これに対するベストプラクティスは自分は持ち合わせてなくて、誰かがいいアイデアを出してくれるだろうと期待していますが、github copilotだけで運用するのであれば、.githubディレクトリでいいのではないかと考えています。ちなみに、今回使ったpromptsディレクトリについては、.agentsディレクトリに配置しても動作しなかったので、.githubディレクトリに配置する必要がありそうでした。
