# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## プロジェクト概要

このリポジトリは Zenn CLI を使用した技術ブログプロジェクトです。技術記事を日本語で執筆・管理しています。

## 基本コマンド

### 開発・プレビュー
```bash
npx zenn preview        # ローカルプレビューサーバー起動
npx zenn new:article    # 新しい記事作成
npx zenn list:articles  # 記事一覧表示
```

### 依存関係管理
```bash
npm install             # 依存関係インストール
npm update              # 依存関係更新
```

## アーキテクチャとディレクトリ構造

```
├── articles/           # 記事ファイル（Markdown + YAML frontmatter）
├── images/            # 記事用画像（記事slugごとにディレクトリ分割）
├── books/             # 本形式コンテンツ（現在未使用）
└── package.json       # Zenn CLI設定
```

## Zenn記事執筆の規約

### フロントマター
```yaml
---
title: "記事タイトル"
emoji: "🧠"
type: "tech"           # "tech" または "idea"
topics: [ai, llm, deeplearning, 初心者]
published: true
published_at: "2024-04-27 12:54"  # 任意
---
```

### 記事構造パターン
- **導入:** `## はじめに` または `## 概要` で開始
- **メッセージボックス:** `:::message` で補足説明
- **詳細セクション:** `:::details タイトル` で詳細内容
- **画像:** `/images/{記事slug}/` に格納、サイズ指定 `=500x` 使用可能

### コードブロック
```markdown
```python:filename.py
コード内容
```
```

### 記事のトーン
- 丁寧で読みやすい日本語
- 専門用語の説明を含む
- 実践的かつ学習過程を共有するスタイル

## 画像管理

### 画像の保存場所
- **推奨方法**: ローカル画像は `/images/{記事slug}/` ディレクトリに格納
  - 記事のslugと同じ名前のディレクトリを作成
  - 例：記事が `d6e44076d3fa0b.md` なら、画像は `/images/d6e44076d3fa0b/` に保存
- Zennアップローダー（Google Cloud Storage）とローカル管理を併用可能

### 画像の参照方法
- **ローカル画像の参照**:
  ```markdown
  ![](/images/d6e44076d3fa0b/1.png)
  ![](/images/d6e44076d3fa0b/screenshot.png =500x)
  ```
- **外部URL画像の参照**:
  ```markdown
  ![](https://storage.googleapis.com/zenn-user-upload/xxxxx.png)
  ```
- サイズ指定: `=500x` や `=300x` を画像URLの後に追加可能

## 記事執筆スタイルガイド

### 記事の基本構成

#### タイトルパターン
- **【技術名】具体的な内容** 形式を採用（例：`【Spring Boot】MockBeanを使った簡単ユニットテスト`）
- 読者が一目で記事の内容を理解できる明確さ
- 技術名をブラケットで強調することで検索性を向上

#### 導入部（はじめに・概要）
- **必須セクション**: すべての記事は `## はじめに` または `## 概要` で開始
- 記事の目的と読者ターゲットを明確化
  - 例：「実務未経験者がテストコードを書いてくれと言われると...」
  - 例：「この記事は、chatGPTなどAIサービスは触ったことがあるけれど...」
- 学習の動機や背景を共有し、読者との共感を生む

#### 環境情報の提示
- 技術記事では使用環境を明示（バージョン、IDE、ツールなど）
```markdown
## 今回の環境
- Spring Boot:3.1.2
- JUnit5
- JDK：Amazon corretto17
```

### 説明スタイル

#### 段階的な理解促進
- **複雑な概念は段階的に説明**
  1. まず全体像やイメージを提示
  2. 詳細な技術解説
  3. 具体的なコード例
  4. 実行結果や動作確認
- 図やイメージを活用して視覚的に理解を助ける
  - 例：システム構成図、処理フロー図、概念図

#### 初心者への配慮
- 専門用語には必ず説明を付ける
- 「要は〜ということ」「イメージとしては〜」など平易な言葉で言い換え
- 「深く考えなくて大丈夫です」「わかりやすくするため」など安心感を与える表現
- 完璧な理解を求めず、「なんとなくイメージを掴む」ことを許容

#### 学習プロセスの共有
- 自分が詰まった点や苦労した部分を率直に共有
- 「初めは〜と思っていましたが」「〜で約2週間かかってしまいました」
- トラブルシューティングの過程を記載することで実践的な学習を促進

### Zenn記法の活用

#### メッセージボックス
```markdown
:::message
重要なポイントや補足説明を記載
:::

:::message alert
警告や注意事項を明示
:::
```
- 本文の流れを妨げずに補足情報を提供
- 読者の注意を引きたい箇所で効果的に使用

#### 詳細セクション
```markdown
:::details タイトル
折りたたみ可能な詳細情報
:::
```
- 補足的な情報や長い説明を格納
- 記事の可読性を保ちつつ、必要な情報を提供

### コード解説のベストプラクティス

#### コードの提示方法
1. **全体像を先に提示**：まずコード全体を示す
2. **部分的に解説**：重要な部分を抜粋して詳しく説明
3. **視覚的な補助**：図やコメントで処理の流れを説明

#### コードブロックの工夫
- ファイル名を明示：` ```java:AccountService.java `
- 重要な行にはコメントで補足
- 複雑な処理は図解と併用

#### AAAパターンなど実践的パターンの紹介
- テストコードでは Arrange、Act、Assert のような業界標準パターンを説明
- パターンを学ぶことで汎用的なスキルを身につけられるよう配慮

### 参考リンクと引用

#### 外部リンクの提示
- 参考にした記事や公式ドキュメントを明記
- 読者がさらに深く学べるよう適切なリンクを提供
- 「詳しくは〜を参照」と学習の道筋を示す

#### 画像の引用
- 他サイトの画像を使用する場合は必ず出典を明記
- 形式：`*https://example.com*` を画像直下に配置

### 記事のまとめ方

#### 結論・まとめセクション
- 学んだ内容の要約
- 記事の要点を簡潔に整理
- 読者へのメッセージ（「誰かの参考になれば幸いです」など）

#### 今後の展望や課題
- リファクタリングの余地や改善点の言及
- 「もう少し〜したい」「今後は〜も検討」など継続的な学習姿勢

### よく使う表現パターン

#### 共感を生む表現
- 「〜ですよね？」「〜だと思います」
- 「少なくとも私はそうでした」
- 「初学者だとよくわからんってなる人も大勢いると思います」

#### 学習を促す表現
- 「わかりやすくするために〜」
- 「イメージだけ掴んだ後、気になったら各自で調べていただくのが良いかと」
- 「まずは〜くらいに考えましょう」

#### 注意喚起の表現
- 「ここで注意すべきは〜」
- 「〜には気をつけてください！」
- 「重要なポイントは〜」

### トピックの選定

#### 実践的な技術記事
- 実際に作成したアプリケーションの解説
- 詰まったポイントとその解決方法
- 新しく学んだ技術の実践的な活用方法

#### よくあるパターン
1. **ライブラリ・フレームワークの使い方**（例：SendGrid、ChakraUI、xml-js）
2. **テストコード作成**（例：MockBeanを使ったユニットテスト）
3. **基本概念の理解**（例：スプレッド構文、LLMの基本）
4. **アプリケーション開発記録**（例：アニメリスト検索アプリ）
5. **クラウドサービスの設定**（例：Azure cosmosDBのロール付与）

### 技術的正確性と誠実さ

#### 不確実性の表明
- 「深い理解はできていなくて、申し訳ないのですが〜」
- 「厳密な定義はわかりませんが〜と解釈しました」
- 「正しい表現ではなさそうな気がするので、あくまでイメージとして」

#### 読者へのフィードバック依頼
- 「何か気づいた点があれば教えていただけると助かります」
- 「間違いがあればご指摘いただけると嬉しいです」
- 「コメントくださると幸いです」

### 記事の長さと密度

- **適度な長さ**: 読み応えがありつつ、読者が疲れない程度
- **情報密度**: 具体的なコード例と説明のバランス
- **分割の検討**: 内容が多い場合は関連記事として分割も検討
  - 例：「詳細は別記事に書いたので、こちらを参考にしてください」

### 記事タイプ別のポイント

#### チュートリアル系記事
- 段階的な手順を明確に示す
- 各ステップの成果物を確認できるようにする
- 躓きやすいポイントを事前に警告

#### 概念説明系記事
- 図やイメージを多用して視覚的に理解を助ける
- 複数の角度から同じ概念を説明
- 「要は〜」という平易な言葉での言い換え

#### トラブルシューティング系記事
- 問題の症状を明確に記述
- 試した解決策の過程を共有
- 最終的な解決方法を明示

### 画像の活用

#### 効果的な画像使用
- システム構成図でアーキテクチャを説明
- 処理フローを図解
- 実際の画面キャプチャで動作を示す
- エラーメッセージのスクリーンショット

#### 画像のサイズ指定
- 適切なサイズ指定で可読性を確保（`=500x`など）
- 大きな図は詳細が見えるサイズで提示

### 継続的改善の姿勢

- 記事公開後も改善の余地を認識
- 「リファクタリングできる部分も色々ありそう」
- 「別の記事を書いてもいいかなとも思っています」
- 読者からのフィードバックを歓迎する態度
